중요한 실행 환경 제약:

- 너는 웹에서 작동한다.
- 사용자 컴퓨터와 `C:\dev\pharmassist-realtime-copilot`에 접근할 수 없다.
- 로컬 경로를 열거나 로컬 파일을 직접 수정했다고 주장하지 마라.
- 업로드된 `repository/`가 현재 프로젝트의 유일한 코드 스냅샷이다.
- 추가 정보가 정말 없을 때만 질문하고, 먼저 첨부 파일을 끝까지 활용하라.
- 결과는 사용자가 로컬 프로젝트에 적용할 수 있는 다운로드 가능한 파일로 반환하라.

목표는 PharmAssist를 “대화를 이어가는 LLM”에서 “검증된 OTC 결정엔진 + 약국별 재고·다빈도 랭킹 + 짧은 문장화 모델”로 재설계하고, 업로드된 스냅샷을 기준으로 구현 가능한 변경 파일을 만드는 것이다.

현재 핵심 결함을 코드로 다시 확인하라.

1. 지식팩은 synthetic 증상 카드 중심이며 제품·성분·공식 claim·source가 런타임에 연결되지 않았다.
2. `RuntimePack`은 사실상 `cards`만 보유한다.
3. 로컬 엔진은 정상 상담도 대부분 `clarify`로 끝낸다.
4. 출력 계약에는 `recommend` 결정 상태가 없다.
5. manifest는 products·claims·sources를 0으로 고정한다.
6. OpenAI가 일반 의약지식으로 빈 데이터를 메우며 구체 성분 출력 요구와 hallucination 차단 규칙이 충돌한다.
7. 결과적으로 “일반의약품을 제시할게요”처럼 행동을 예고하고 실제 성분·제품 없이 끝날 수 있다.

프롬프트 문구만 더 추가하는 방식으로 해결하지 마라. 전체 약 데이터를 LLM context에 넣는 방식도 금지한다. 전체 제품 registry, 약사 검토 formulary, 상담별 retrieved context를 분리하라.

`RecommendationDecision.status`는 최소 `recommend | ask | refer | insufficient`를 지원해야 한다. `recommend`이면 검증된 ingredient option이 필수다. 재고가 연결된 tenant에서는 product candidate도 필수다. `ask`는 실제 선택을 바꾸는 질문 하나만 허용한다. `refer`에서는 제품 후보를 출력하지 않는다. 모든 추천은 활성 지식팩의 ID와 source ref로 검증되어야 한다.

다빈도 제품은 전국 임의 top-N으로 정하지 마라.

1. 공급실적 있는 OTC 모집단
2. 허가취하·단종·DUR 차단 제거
3. tenant POS의 최근 90일 판매를 공식 product ID에 crosswalk
4. 누적 판매 85~90% coverage 후보 계산
5. 증상 카테고리별 최소 coverage 보정
6. 약사 승인 후 tenant formulary 활성화
7. 런타임 정렬은 임상 적합성 → 안전 → 재고 → 다빈도 순서

판매량이나 마진이 임상 적합성 점수를 바꾸면 안 된다. 재고·다빈도는 안전하고 동등한 제품 사이의 후순위 랭킹에만 쓴다.

필수 변경 영역:

- contracts: 새 엔터티·출력 JSON Schema·generated types·validator
- database: authoring·review·formulary·pack metadata migration
- tools/ingest: provider adapter·source snapshot·crosswalk·lint
- knowledge: source → claim → protocol → review → compile → sign
- retrieval: protocol·ingredient·product index
- 새 recommendation package: eligibility·exclusion·ingredient selection·inventory mapping·ranking
- runtime: ConsultationState → safety gate → protocol → RecommendationDecision
- API: 실제 manifest counts·pack activation·formulary import·tenant isolation
- OpenAI adapter: 고정된 decision을 한 문장으로 표현만 하고 새 제품·성분 생성 차단
- web: 결정 상태·성분·제품 후보·재고·근거 표시

현재 스냅샷에는 다른 작업의 미커밋 변경이 포함될 수 있다. 기존 변경을 되돌리거나 통째로 교체하지 마라. 특히 UI와 `packages/openai-adapter`의 최신 의도를 보존하고, 변경 파일마다 기존 코드와 어떻게 통합했는지 `PATCH_MANIFEST.md`에 써라.

필수 검증 기준:

- “제시할게요”, “안내할게요”, “확인해볼게요”로 끝나는 미완성 응답 0건
- `recommend`인데 성분 후보가 없는 결과 0건
- 활성 pack에 없는 성분·제품 출력 0건
- source ref 없는 추천 0건
- red flag·금기 상황에서 제품 추천 0건
- 한 turn 질문 최대 1개, 이미 답한 질문 반복 0건
- 재고 0·비활성·단종 제품 추천 0건
- LLM·네트워크 장애에도 로컬 결정 유지
- synthetic·미검토·만료 pack production 활성화 차단
- 기존 privacy/security/safety 회귀 없음

기침, 콧물, 인후통, 속쓰림, 소화불량, 설사·변비, 근육통, 피부 증상, 발열 대화를 golden test에 포함하라. 구체 임상 선택 규칙은 공식 근거와 약사 승인 없이는 production claim으로 만들지 마라. 자동 추출물은 candidate 상태로만 둬라.

최종 결과는 `PharmAssist-Pro-Result.zip` 하나로 반환하라. 구조:

```text
RESULT_SUMMARY.md
```

`changed-files/`에는 업로드된 `repository/` 기준 상대경로를 그대로 유지한 수정·신규 파일을 넣어라. 예를 들어 `repository/app/packages/runtime/src/index.ts` 수정본은 `changed-files/app/packages/runtime/src/index.ts`에 둔다. 부분 코드 조각만 채팅에 흩뿌리지 마라.

웹 환경에서 실제 테스트를 실행할 수 있으면 사용한 명령과 원문 결과를 남겨라. 실행할 수 없으면 통과했다고 쓰지 말고 `VALIDATION_PLAN.md`에 로컬 실행 명령과 미검증 상태를 명시하라. 사용자 컴퓨터에 적용·commit·push·release·deploy했다고 주장하지 마라. 임상 production-ready라고 과장하지 마라.
