# 01. Product Requirements Document

## 1. 문제 정의

약국 카운터 상담은 짧고 동시다발적이다. 환자는 완전한 병력 대신 “기침약 주세요”, “아이 몇 mL예요?”, “피임약 빼먹었어요”처럼 불완전한 문장을 말한다. 약사는 안전을 위해 추가 질문을 해야 하지만, 다양한 OTC·처방약·특수군·상호작용·의뢰 기준을 즉시 떠올려야 한다.

본 제품은 지식을 길게 설명하는 챗봇이 아니라, 약사가 다음 1~2문장을 신속히 결정하도록 돕는 **glanceable counseling copilot**이다.

## 2. 사용자와 이해관계자

### Primary user

- 대한민국 약국의 면허 약사
- 데스크톱/노트북/태블릿 브라우저 사용
- 한 손은 조제·POS·제품 확인에 사용 중일 수 있음
- 입력에 2~8개 단어만 쓸 가능성이 높음

### Secondary stakeholders

- 약국 관리자/지식 관리자
- 임상 검수 약사
- 개인정보 보호/법무 담당자
- 개발·운영 담당자
- 환자: 직접 사용자는 아니지만 안전·고지·프라이버시의 주체

## 3. 핵심 사용자 시나리오

### 시나리오 A — 키워드 입력

1. 약사가 `/` 또는 검색창에 `4세 해열제 몇 ml` 입력
2. 250ms 안에 “체중·제품·농도 확인 전 숫자 금지” 카드가 뜸
3. 화면에 다음 질문 하나가 크게 표시됨
4. 약사가 체중/제품을 입력하면 검증된 제품 규칙이 있을 때만 계산 후보가 표시됨
5. 고위험 결과는 약사 확인을 거쳐 발화

### 시나리오 B — 환자 발화 듣기

1. 약사가 push-to-talk를 누르고 환자에게 음성 처리 사실을 고지
2. Realtime transcription delta가 도착
3. `피임약 ... 하루 빼먹...` prefix로 잠정 카드 표시
4. 환자가 제품명/누락 기간을 말하면 카드가 안정화
5. 약사가 카드를 freeze한 뒤 환자와 대화

### 시나리오 C — 위험 신호

1. `입술 붓고 숨이 안 쉬어져요` 입력/전사
2. debounce를 건너뛰고 응급 카드가 최상단에 고정
3. 일반약/제품 추천을 숨김
4. “즉시 응급평가” 행동만 표시
5. 약사가 확인 버튼을 누르기 전 다른 결과가 덮어쓰지 못함

### 시나리오 D — 오프라인/장애

1. OpenAI 또는 인터넷 연결 실패
2. 타이핑·로컬 카드·안전 규칙은 정상 동작
3. 음성/문장 보정 기능만 `제한됨` 배지
4. 원래 카드가 사라지거나 빈 화면이 되지 않음

## 4. 기능 요구사항

### FR-01 입력

- 타이핑, 붙여넣기, 빠른 슬롯 버튼, 키보드 단축키를 지원한다.
- 음성은 기본 push-to-talk이며 명시적 활성화가 필요하다.
- 한국어 띄어쓰기 변형, 단위 표현, 숫자, 일반적인 약국 약칭과 ASR 오인식을 정규화한다.
- 입력 중 이름·전화번호·주소 등 PII를 외부 API 전 제거하거나 마스킹한다.

### FR-02 부분 입력 처리

- `voice_partial`과 `voice_final`을 구분한다.
- 부분 입력의 카드는 `provisional`로 표시한다.
- 안정 prefix 또는 높은 exact score가 생기면 즉시 카드화한다.
- 현재 카드보다 명확히 높은 점수이거나 위험신호가 아닌 한 화면을 자주 바꾸지 않는다.

### FR-03 의도/카드 검색

검색 순서는 고정한다.

1. critical red-flag lexical/rule gate
2. exact alias lookup
3. structured slot/rule match
4. local BM25/FTS
5. fuzzy n-gram 후보
6. 모호하면 상위 3개 선택 UI
7. 선택적으로 LLM 보정

LLM을 의도 검색의 필수 단계로 두지 않는다.

### FR-04 출력

항상 구조화된 형태로 반환한다.

- `say_now`: 최대 3개, 첫 항목이 가장 중요
- `ask_next`: 기본 1개, 최대 3개
- `red_flags`
- `actions`
- `avoid`
- `missing_slots`
- `confidence`, `candidate_intents`
- `source_refs`, `knowledge_version`, `latency`

### FR-05 안전

- 임상 위험신호가 있으면 추천 경로를 중단한다.
- 필수 슬롯이 없으면 차단형 질문 카드를 출력한다.
- 제품별 데이터 없이 제품별 규칙을 추정하지 않는다.
- 약물 식별은 포장/처방/공식 제품정보 없이 외형만으로 하지 않는다.
- 고위험 action에는 `requires_confirmation=true`를 둔다.

### FR-06 출처·검수

- 모든 운영 claim은 출처, locator, trust tier, 검증일, 만료일, 검수자를 가진다.
- C/D/X 첨부 자료만으로 운영 claim을 만들 수 없다.
- 카드 클릭 시 약사가 상세 근거를 열 수 있으나 첫 화면에는 요약만 보인다.
- 만료되거나 충돌 상태인 claim을 참조하는 카드는 자동 비활성화한다.

### FR-07 피드백

- 약사는 `사용`, `수정`, `거절`, `의뢰 전환`을 빠르게 기록할 수 있다.
- 수정 사유는 코드형으로 저장하고 자유 텍스트는 기본 수집하지 않는다.
- 환자 식별자나 원문 대화와 피드백을 결합하지 않는다.

### FR-08 관리자/저작

- 출처 등록, claim 추출, 충돌 표시, 약사 검수, 만료 관리, knowledge pack 배포/롤백을 지원한다.
- 자동 추출 결과는 언제나 draft/candidate다.
- source→claim→card 추적 가능해야 한다.

## 5. 비기능 요구사항

### 성능

- 브라우저 로컬 exact/rule P95 ≤ 250ms
- local fuzzy P95 ≤ 400ms
- 안정 음성 prefix 이후 P95 ≤ 700ms
- UI commit P95 ≤ 16ms
- LLM 보정 timeout 기본 2.5초; 실패 시 로컬 카드를 유지
- app cold start에서 compact pack 로드 P95 ≤ 1.5초, 이후 offline 사용 가능

### 가용성

- OpenAI 장애가 typed core를 중단시키지 않는다.
- production knowledge pack은 이전 2개 버전을 로컬에 보관해 즉시 rollback한다.
- 서버가 응답하지 않아도 현재 카드와 로컬 검색은 유지한다.

### 보안/프라이버시

- 브라우저에 표준 OpenAI API key를 노출하지 않는다.
- Realtime은 서버가 생성한 제한된 세션 또는 unified WebRTC 경로를 사용한다.
- Responses는 `store:false`; 지속 conversation 객체는 기본 금지.
- 원문 음성/전사는 기본 저장하지 않는다.
- 로그에는 카드 ID, 룰 ID, latency, 결과 코드만 기록한다.

### 접근성/사용성

- 카운터 거리에서 읽을 수 있는 글자와 대비
- 마우스 없이 전 기능 사용 가능
- 색상 외 텍스트/아이콘으로 위험도 표시
- 화면 깜빡임/자동 스크롤 최소화
- 한국어 IME composition 중 조기 검색으로 입력을 깨뜨리지 않음

## 6. 비목표

- 환자에게 직접 음성으로 답하는 자율 상담사
- 질환 진단 또는 처방 변경
- 약사 승인 없는 자동 제품 판매
- 전체 첨부 파일을 그대로 vector store에 넣는 런타임 RAG
- 모델의 기억에 용량·금기 규칙을 맡기는 방식
- 사람/동물/건기식 지식을 한 검색공간에 섞는 방식
- 모든 약국 시스템과 첫 버전부터 통합

## 7. 제품 범위와 모드

### v1 기본 모드

`human_otc`만 활성화한다. 처방 복약지도·건기식·동물약은 UI와 인덱스 차원에서 분리한다.

### 처방 복약지도 모드

정확한 제품/처방 항목이 제공될 때만 동작한다. 임의 발화만으로 처방약을 추정하지 않는다.

### 건기식 모드

질환 치료를 약속하지 않으며, 의약품 상호작용·임부·간/신장 위험을 먼저 확인한다. 첨부 자료만으로 자동 활성화하지 않는다.

### 동물약 모드

사람 모드와 별도 앱/패키지 수준의 분리를 권장한다. 수의학 검수 없이는 구현하지 않는다.

## 8. 우선순위

1. 임상 안전
2. 개인정보 보호
3. 출처 정확성·최신성
4. 첫 카드 지연시간
5. 약사의 한눈 사용성
6. 장애 복원력
7. 비용
8. 기능 수

## 9. 완료 정의

기능이 화면에 존재하는 것만으로 완료가 아니다. 각 요구사항에 unit/integration/e2e/benchmark 테스트가 있고, 합성 data만으로 전체 플로가 실행되며, production data gate가 미승인 지식을 차단해야 한다.
